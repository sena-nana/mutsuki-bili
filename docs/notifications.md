# 推送功能

插件通过定时轮询 B 站 API 检测状态变化，并将通知推送到已订阅的聊天频道。

## 直播推送

### 检测逻辑

1. 按 `liveInterval` 间隔轮询所有订阅了 `live` 类型的 UID
2. 获取 UP 主用户信息，取得直播间 ID
3. 查询直播间状态（`live_status`：0=未播，1=直播中，2=轮播）
4. 与 `bili.live_state` 表中缓存的上次状态比较：
   - 未播 → 直播中：触发**开播通知**
   - 直播中 → 未播：触发**下播通知**

### 消息格式

**开播通知：**

```
[封面图片]
【开播】UP主名称
标题：直播间标题
分区：直播分区名
https://live.bilibili.com/房间号
```

**下播通知：**

```
【下播】UP主名称 的直播结束了
```

## 动态推送

### 检测逻辑

1. 按 `dynamicInterval` 间隔轮询所有订阅了 `dynamic` 类型的 UID
2. 获取 UP 主最新动态列表
3. 与 `bili.dynamic_state` 表中的 `lastDynamicId` 比较（使用 BigInt 确保高精度 ID 比较正确）
4. 筛选出所有 ID 大于上次记录的新动态
5. 按时间正序（从旧到新）逐条推送

### 首次检测

首次轮询某个 UID 时（`lastDynamicId` 为 `"0"`），插件只记录最新的动态 ID 而不发送通知，避免补发大量历史动态。

### 支持的动态类型

- **图文动态**（opus / draw）：文字 + 图片
- **视频动态**（archive）：包含视频封面、标题和链接
- **纯文字动态**：仅包含文字内容

### 消息格式

```
【动态】UP主名称
动态文字内容
[图片1]
[图片2]
https://t.bilibili.com/动态ID
```

带视频的动态：

```
【动态】UP主名称
动态文字内容
[视频封面图]
视频：视频标题
https://www.bilibili.com/video/BVxxx
```

## 视频推送

### 检测逻辑

1. 按 `videoInterval` 间隔轮询所有订阅了 `video` 类型的 UID
2. 通过 WBI 签名的 API 获取 UP 主最新 5 个视频
3. 与 `bili.video_state` 表中的 `lastBvid` 比较
4. 筛选出发布时间晚于上次记录的新视频
5. 按时间正序（从旧到新）逐条推送

### 首次检测

与动态推送类似，首次轮询时（`lastBvid` 为空）只记录最新视频 ID 而不发送通知。

### 消息格式

```
[封面图片]
【投稿】UP主名称
视频标题
https://www.bilibili.com/video/BVxxx
```

## 轮询策略

### 多 UID 间隔

当监控多个 UP 主时，插件在轮询每个 UID 之间插入 **2 秒延迟**，防止短时间内大量请求触发 B 站限速。

### 三类轮询独立运行

直播、动态、视频三个轮询循环互相独立，按各自的间隔并行运行：

| 类型 | 默认间隔 | API 接口 |
|------|----------|----------|
| 直播 | 1 分钟 | `x/space/acc/info` + `room/v1/Room/get_info` |
| 动态 | 2 分钟 | `x/polymer/web-dynamic/v1/feed/space` |
| 视频 | 5 分钟 | `x/space/wbi/arc/search`（WBI 签名） |

### 重试与容错

- 请求失败后自动重试，使用指数退避策略（默认最多 3 次）
- 触发 HTTP 429 限速时，等待 `rateLimitBackoff`（默认 5 分钟）后重试
- Cookie 过期（错误码 -101 / -111）时记录警告日志，不再重试
- 单个 UID 的失败不影响其他 UID 的轮询

### 消息分发流程

1. 从 `bili.admin` 表查询该 UID 的所有订阅记录
2. 过滤条件：包含对应推送类型 + 未暂停（`paused = false`）
3. 解析 `channel` 字段获取目标平台和频道 ID
4. 查找对应平台的在线 Bot 实例
5. 调用 `bot.sendMessage()` 发送格式化的通知消息
