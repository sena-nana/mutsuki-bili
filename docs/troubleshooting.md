# 故障排除

## Cookie 过期

**现象：** 日志中出现 `Cookie 可能已过期 (code=-101)` 或 `code=-111` 的警告。

**原因：** B 站的登录凭证已过期或失效。

**解决方法：**
- 通过控制台「哔哩哔哩」页面重新扫码登录
- 或在聊天中使用 `bili.login` 命令重新登录
- 如果是手动配置的 Cookie，需要从浏览器重新获取

---

## 触发限速

**现象：** 日志中出现 `触发限速，等待 Xs` 的提示。

**原因：** 短时间内对 B 站 API 的请求次数过多，收到 HTTP 429 响应。

**解决方法：**
- 插件会自动等待 `rateLimitBackoff`（默认 5 分钟）后重试
- 如果频繁触发，建议增大轮询间隔（`liveInterval`、`dynamicInterval`、`videoInterval`）
- 减少监控的 UP 主数量也能降低请求频率

---

## 推送没有收到

**排查步骤：**

1. **检查 Bot 是否在线** — 确认对应平台的 Bot 处于连接状态
2. **检查 channel 格式** — `bili.admin` 表中的 `channel` 字段是否为正确的 `platform:channelId` 格式
3. **检查推送类型** — 确认 `types` 字段包含了期望的通知类型（`live`、`dynamic`、`video`）
4. **检查暂停状态** — 确认 `paused` 字段为 `false`，可使用 `bili.admin.list` 查看
5. **检查日志** — 查看 Koishi 日志中是否有 `mutsuki-bili` 相关的错误或警告信息
6. **检查 UID 是否正确** — 确认绑定的 B 站 UID 正确且对应的 UP 主存在

---

## 首次添加 UP 主没有收到历史消息

**这是预期行为。** 首次轮询某个 UID 时，插件只会记录当前最新的动态 ID / 视频 BV 号作为基准点，不会补发历史内容。后续新发布的内容才会触发通知。

如果需要确认推送功能正常，可以使用 `bili.preview` 命令发送测试推送。

---

## 控制台「哔哩哔哩」页面不显示

**排查步骤：**

1. 确认已安装 `@koishijs/plugin-console`（>= 5.30.0）
2. 确认 `@koishijs/plugin-console` 已启用
3. 确认 `mutsuki-bili` 插件已正常加载（检查日志中是否有启动成功的提示）
4. 尝试刷新控制台页面

---

## 扫码登录失败

**二维码过期：** 二维码有效期约 3 分钟，过期后需要重新执行 `bili.login` 或在控制台重新获取。

**登录后仍然提示未登录：** 可能是配置文件中填写了空的 `sessdata`。插件会优先检查配置文件的 Cookie，确认配置文件中没有留下空字符串或无效值。

---

## 请求失败并频繁重试

**现象：** 日志中反复出现 `请求失败（已重试 N 次）` 的警告。

**可能原因：**
- 网络连接不稳定
- B 站 API 接口发生变更
- 部署环境无法访问 B 站 API（如海外服务器被限制）

**解决方法：**
- 检查服务器网络连通性：确认能正常访问 `api.bilibili.com`
- 检查是否需要配置代理
- 更新到最新版本插件
